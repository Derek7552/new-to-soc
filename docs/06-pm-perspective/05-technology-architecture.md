# SOC 产品技术架构决策

> 面向：产品经理、技术负责人、CTO、架构师 — 需要理解"做 SOC 产品的关键技术决策有哪些、不同选择对产品形态和商业策略的影响"
> 解决的核心决策问题：技术架构选什么？（数据湖 vs 联邦查询 / 构建 vs 购买 / 开放 vs 封闭）

## TL;DR

SOC 产品的技术架构决定了商业天花板。三个最关键的架构决策：**数据策略**（自己存数据湖，还是 API 现取现用）、**AI 策略**（LLM 嫁接还是 AI 原生）、**集成策略**（深度对接少量平台，还是广度覆盖）。这些决策直接影响目标客户群（中大 vs 中小）、部署周期（数月 vs 1 小时）、定价模型（按数据量 vs 按使用量）和竞争壁垒。对 PM 而言，不理解这些技术决策的商业后果，就无法做出正确的产品定位。

## 决策 1：数据策略 — 数据湖 vs 联邦查询

这是 SOC 产品最根本的技术分歧，决定了产品形态、部署成本和目标客户。

### 方案 A：自建数据湖（传统平台路线）

```
客户环境 → 各种数据源 → 日志采集管道 → 你的数据湖 → 检测引擎 → 告警
                                    ↑
                              你可控的分析环境
```

**代表**：Splunk、Cortex XSIAM、Elastic Security、Microsoft Sentinel

**商业含义**：

| 维度 | 影响 |
|------|------|
| **目标客户** | 数据量大（TB/天级别）的大型企业——他们本来就需要存日志 |
| **部署周期** | 数周到数月（调试日志管道、存储规划、索引优化） |
| **收入模式** | 按数据摄入量收费 → 收入与客户数据增长线性挂钩 |
| **迁移成本** | 极高 → 客户粘性强 → 竞争壁垒高 |
| **毛利结构** | 存储成本占比高 → 规模效应有递减 |
| **技术壁垒** | 高（海量日志的实时索引和查询是极难的工程问题） |

### 方案 B：联邦查询（AI 原生路线）

```
客户环境 → AI Agent → 通过 API 实时查询客户已有的 SIEM/EDR/IAM/...
                      ↑
                不存数据，现取现用
```

**代表**：Dropzone AI、部分 Torq 功能

**商业含义**：

| 维度 | 影响 |
|------|------|
| **目标客户** | 中型企业、已有 SIEM/EDR 但缺人力的组织 |
| **部署周期** | 数小时到 1 天（只接 API，不需要建数据管道） |
| **收入模式** | 按使用量（调查次数）收费 |
| **迁移成本** | 低 → 客户容易切换 → 需要靠产品体验和准确率保持粘性 |
| **毛利结构** | 极高（无存储成本，只有 LLM API 调用成本） |
| **技术壁垒** | 中等（取决于 API 集成的广度和 AI 推理的准确度） |

### 两种路线的取舍

| 选择数据湖路线 | 选择联邦查询路线 |
|---------------|-----------------|
| 能做全栈一体化，客单价高 | 轻资产 = 高毛利，部署快 = 销售周期短 |
| 但排除中小客户（用不起） | 但依赖底层 SIEM/EDR 的数据质量 |
| 客户粘性高（迁移成本高） | 客户容易流失（迁移成本低） |

### 混合路线：趋势所在

越来越多产品采用"不存全部，但缓存关键上下文"的混合策略：
- 全量日志留存在客户现有 SIEM 中
- AI Agent 的调查结果、证据链、推理过程缓存在自己平台
- **好处**：既有轻部署的优势，又有持续积累的数据资产

→ **PM 判断框架**：问自己"我的目标客户是已经有数据湖的，还是连数据湖都没有的？"——答案决定了你的数据策略。

## 决策 2：AI 策略 — 嫁接 LLM vs AI 原生架构

### 方案 A：LLM 嫁接（大多数传统厂商的做法）

在已有平台的产品上，用 LLM 做：
- 告警摘要生成
- 自然语言查询转 SPL/KQL
- 风险评分解释
- 推荐 Playbook

**商业含义**：
- 上市时间快（6-12 个月可交付第一版 AI 功能）
- 核心架构不变——分析师仍然需要自己调查
- 容易被竞品"我也加了 AI"抹平差异化
- **最大的风险**：底层架构不是为 AI 设计的，后续扩展会撞天花板

### 方案 B：AI 原生架构（创业公司的主流选择）

从架构起点就以 AI Agent 为核心：
- 系统按"AI 工种"（Analyst / Hunter / Intel）组织，而非"工具模块"
- 每个 AI Agent 有独立的上下文窗口、工具调用链、推理记忆
- 数据模型设计时优先考虑 AI 可读性，其次才是人类可读性

**商业含义**：
- 开发周期更长，但架构灵活度更高
- 适合从零搭建的新产品
- 差异化更强（"AI 原生"本身就是市场信号）
- **最大的风险**：如果 LLM 技术路线突变，整个产品可能需要重构

### 两种路线的实际表现对比

| 场景 | LLM 嫁接 | AI 原生 |
|------|---------|---------|
| 应对一种新攻击 | 需要先有人写检测逻辑 | AI 能从少量标注样本中泛化 |
| 解释为什么是误报 | 生成一段摘要（可能是编的） | 展示每一步实际的工具调用和查询结果 |
| 分析师纠正 AI | 无法闭环（需要工程师改规则） | 自然语言纠偏，持续学习 |
| 多步骤调查 | 每步需要人点击"下一步" | AI 自主完成全流程 |

→ **PM 判断框架**：不要纠结术语——问你的工程团队三个问题：
1. "AI 的每一步调查动作能不能被日志记录下来？"（可解释性）
2. "分析师纠正 AI 后，AI 下次能不能变好？"（学习闭环）
3. "如果换成另一个 LLM 供应商，架构能不能适应？"（模型无关性）

## 决策 3：集成策略 — 深度 vs 广度

### 广度策略（覆盖 90+ 产品）

**代表**：Dropzone AI（90+ 原生集成）、Cortex XSOAR（1100+ 集成）

**做法**：对每个集成做基础对接——查询告警、获取原始日志、执行简单操作。

**商业含义**：
- 客户覆盖面广，销售时"你用的工具我们对接了"
- 但集成深度有限——很多集成只是"能调 API 拿到数据"而非"理解了数据语义"
- 维护成本高——每个被集成的产品升级 API，你都要跟着改

### 深度策略（深耕 5-10 个核心平台）

**做法**：对 Splunk、CrowdStrike、Sentinel、Okta 等几个最主流的平台做极深度集成——不只是调 API，而是理解其数据模型、查询语言、上下文结构。

**商业含义**：
- 在支持的核心平台上体验极佳
- 但客户用了不支持的平台你就覆盖不到
- 维护成本聚焦

### 设计好的集成策略

1. **对核心平台深度集成**：每个品类选 1-2 个市场占有率最高的（如 EDR 选 CrowdStrike + Defender）
2. **对其他平台提供标准适配器**：通过通用 API 框架支持长尾产品
3. **让客户自己扩展**：开放集成框架，允许客户或 MSSP 自己写适配器

## 决策 4：部署模式 — SaaS vs 本地化 vs 混合

| 部署模式 | 适用市场 | 产品架构影响 |
|---------|---------|-------------|
| **纯 SaaS** | 欧美中型企业、MSSP | 架构最简单，多租户，更新快 |
| **本地化部署** | 中国政企、金融、军工 | 架构最复杂，需要离线运行 AI 推理 |
| **混合（SaaS 控制面 + 本地 Agent）** | 大型企业、跨国机构 | 数据留本地，AI 控制在云端 |

PM 需要注意：
- **中国市场的本地化部署是硬需求**：很多政企客户不允许数据出境/出机房
- 如果做本地化，LLM 推理必须在本地跑（或通过私有化部署的模型）——这意味着不能用"调 OpenAI API"的简单方案
- 混合部署的复杂度比纯 SaaS 高一个数量级，团队规模和交付能力要匹配

## 决策 5：开放 vs 封闭生态

### 封闭生态

**代表**：Palo Alto Cortex（全栈一体化，数据湖锁定）

**商业逻辑**：客户用得越多，迁移越难，LTV 越高

**风险**：客户越来越反感"锁定"，监管也可能介入（类似欧盟对 iOS 封闭生态的监管）

### 开放生态

**做法**：
- API-first 设计，所有能力可被外部调用
- 支持 MCP（Model Context Protocol），允许第三方 AI Agent 协作
- 数据可导出，查询语言尽量兼容标准（如 SPL、KQL）

**商业逻辑**：融入客户已有的安全工具链，做"最好的那一层"而非"唯一的那一个"

### 判断框架

- **如果你在从零做新产品**：开放是更好的选择——降低客户试错门槛
- **如果你已经是大厂/领先者**：封闭最大化收入，但要控制客户"被锁定"的负面感知
- **MCP 可能改变游戏规则**：如果 MCP 成为行业标准，"开放 API + 支持 MCP"就是入场券而不是差异化

## PM 必知：三个技术概念

### 1. 日志解析器（Parser）

不同厂商的日志格式不同（防火墙的日志和 Windows 事件日志完全不一样）。Parser 是把原始日志转换为统一结构化数据（字段名、时间戳、事件类型）的组件。

**商业含义**：每支持一个新数据源，就可能需要写一个新 Parser。100 个数据源 × 每个需要 1-2 周开发 → 工程投入巨大。集成广度 vs Parser 质量的权衡是 SOC 产品的日常工程难题。

### 2. 检测规则 / 检测工程

传统 SOC 用"规则"检测威胁（如"30 分钟内 10 次登录失败 → 告警"）。AI 原生 SOC 则用行为基线 + 异常检测。

**商业含义**：
- 规则路线：客户可以自己写（灵活），但误报率高（需要大量调优工作）
- AI 路线：开箱误报少（体验好），但客户没法自己改（受限）
- 混合路线是主流：规则覆盖已知威胁，AI 发现未知异常

### 3. 数据驻留与合规

SOC 平台处理的日志可能含 PII（个人身份信息）、受监管行业数据（PCI、HIPAA）。不同国家对数据存储位置有法律要求。

**商业含义**：如果你的产品存数据，必须支持按国家/区域指定存储位置。这是金融、医疗、政府客户的准入条件——不是 nice-to-have。

## 对产品决策者的关键启示

1. **技术架构 = 商业策略**：数据湖 vs 联邦查询不只是技术选型，直接决定了你的目标客户、定价和竞争壁垒
2. **先确定目标客户**，再选架构：中小客户不需要数据湖（他们用 SaaS SIEM），大客户不接受联邦查询（他们需要数据掌控权）
3. **MCP 值得从产品第一天就支持**：它是 AI Agent 互操作的开放标准，支持它 = 降低被替代风险 + 降低客户集成成本
4. **Parser 生态是沉默的工程负债**：支持 90+ 集成听起来好，但维护起来是持续的成本中心
