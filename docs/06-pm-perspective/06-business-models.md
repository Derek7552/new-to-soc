# SOC 产品商业模式与定价

> 面向：产品经理、业务负责人、CEO、商业化团队 — 需要理解"SOC 产品怎么赚钱、不同定价模型对产品行为的影响"
> 解决的核心决策问题：怎么定价？怎么变现？用什么商业模式？

## TL;DR

SOC 产品有四种主流定价模型：按数据量、按用户/资产数、按使用量（调查次数）、按服务人天。定价模型不是事后商业策略——它从产品设计第一天就塑造用户行为。按数据量定价会让客户"少接日志"（降低安全可见度），按调查次数定价会让客户"挑拣告警"（可能漏掉真阳性）。好的定价模型与客户利益对齐：客户越安全，你越赚钱。

## 四种主流定价模型

### 模型 1：按数据摄入量（GB/TB per Day）

**代表**：Splunk、Cortex XSIAM、大多数 SIEM

**怎么算**：
```
月费 = 每日数据摄入量（GB/天）× 单价 × 数据留存天数
```

**对产品行为的影响**：
- 客户会主动减少数据摄入（少接日志源、降采样）→ **降低安全可见度**
- 安全团队要和预算博弈："这条日志值得花 $X/GB 接入吗？"
- 厂商收入随客户数据增长而增长——但客户数据增长 ≠ 客户获得更多价值

**对 PM 的启示**：如果你是 SIEM 厂商，你的定价在**惩罚好行为**（客户越认真做安全，日志越多，越贵）。这是这个品类最大的商业设计缺陷。

### 模型 2：按用户/资产数

**代表**：CrowdStrike（按终端数）、Microsoft Sentinel（部分按用户）

**怎么算**：
```
年费 = 受保护终端/用户数 × 每终端/用户单价 × 模块数
```

**对产品行为的影响**：
- 可预测性强——客户预算稳定，厂商收入稳定
- 但客户会"部分部署"（只给关键服务器装 Agent）
- Agent 覆盖率不是 100% → 安全盲区 → 出了事客户可能"但我只部署了 30%"

**对 PM 的启示**：这个模型相对健康——收入与客户资产规模关联，不与安全事件数量关联。但"部分部署"导致的安全盲区是隐性风险。

### 模型 3：按使用量（调查次数 / Playbook 执行次数 / API 调用量）

**代表**：Dropzone AI（按调查次数）、部分 SOAR 产品

**怎么算**：
```
年费 = 基础费 + 超额使用量 × 单价
或：年费 = 含 X 次调查，超出按 Y/次计
```

**对产品行为的影响**：
- 客户每发一条告警给 AI 调查都有成本 → 可能"挑拣"高优先级才送
- 低优先级告警中的真阳性可能被漏掉（因为没让 AI 查）
- 但计费简单、容易算 ROI（"一次 AI 调查 $9，一个分析师调查同一条告警 $X0"）

**对 PM 的启示**：这个模型的问题是**定价在惩罚使用**。解决方向是按"资产数"而非"调查次数"定价（Dropzone 似乎也在往这个方向调整）。

### 模型 4：按服务人天 / 托管服务

**代表**：MDR 厂商、MSSP

**怎么算**：
```
年费 = 7×24 值守 × 分析师等级 × 服务内容
```

**对产品行为的影响**：
- 人是成本中心，毛利率取决于分析师利用率
- 需要用 AI/自动化工具降低人力密度
- MSSP 客户粘性极高（换 MSSP = 重新建立信任 + 重做集成）

**对 PM 的启示**：如果你在平台产品上叠加服务收入，MSSP 模式是最高 LTV 的路径——但技能密集（招分析师），扩张速度受限。

## 定价模型对比：一张表看穿

| 模型 | 客户可预测性 | 厂商收入可预测性 | 与客户利益是否一致 | 主要副作用 |
|------|-------------|----------------|-------------------|-----------|
| 按数据量 | 低（数据量难预测） | 中 | **不一致** — 客户减少日志 = 厂商收入下降 | 惩罚安全可见度 |
| 按用户/资产 | 高（资产数相对稳定） | 高 | **一致** — 资产增多 = 安全需求增多 | 客户可能部分部署 |
| 按使用量 | 中 | 中 | **部分一致** — 但客户可能省着用 | 惩罚使用 = 漏掉威胁 |
| 按服务人天 | 高（固定合同） | 高 | **一致** — 客户越安全服务效率越高 | 人力扩张受限 |

## 商业模式创新案例

### Dropzone AI：预付费套餐 + 内置情报订阅

```
基础定价：$36,000/年
含：4,000 次 AI 调查 + $18K+ 内置威胁情报订阅（CrowdStrike Falcon Intel、GreyNoise）
```

**商业设计亮点**：
- 打包情报订阅解决了"客户还要另外买情报"的痛点——一单解决两个采购需求
- 4000 次调查 = 约 11 次/天——对中小 SOC 来说足够，对大 SOC 需要扩展
- 内置情报订阅 $18K 的价值锚定让 $36K 显得"超值"

### CrowdStrike：模块化交叉销售

```
Falcon 终端（必须）→ + Falcon Insight（EDR）→ + Falcon Overwatch（猎人）→ 
+ Falcon Complete（MDR）→ + Charlotte AI（AI 助手）
```

**商业设计亮点**：
- 终端 Agent 是"地基"——客户一旦部署，加模块的边际成本极低
- NDR（Net Dollar Retention）极高：现有客户每年加模块，收入持续增长
- Agent 部署后迁移成本极高 → 续费率接近 100%

### Microsoft：防御性免费

```
Defender for Endpoint 绑定 M365 E5 → 客户已经是 Microsoft 全家桶 → 
安全功能"免费"附带 → 纯安全厂商被迫与"零价格"竞争
```

**商业设计亮点**：
- 不做安全也能赚钱（靠 M365 授权费）
- 安全是"免费"附赠——纯安全厂商无法匹敌这个价格
- 策略上这是防御性的：不是为了抢安全市场，而是防止客户因为安全需求换到别的生态

## PM 定价设计自检清单

设计或审视你的 SOC 产品定价时，逐条检查：

- [ ] **定价与客户利益是否对齐？** 客户越安全 → 你赚得越多（或至少不减少）？
- [ ] **是否有"惩罚好行为"的副作用？** 按数据量定价惩罚"多接日志"，按调查次数定价惩罚"多调查"
- [ ] **定价是否可预测？** 客户能提前知道下月费用吗？还是每次收到账单都"surprise"？
- [ ] **起步价格是否排除核心目标客户？** 如果目标是中型企业，$100K/年的起步费等于不卖给他们
- [ ] **是否有自然扩展路径？** 客户用了一年之后，怎么买更多？（更多资产？更多模块？更高级别？）
- [ ] **试用门槛够不够低？** SOC 产品采购决策风险极高——有没有办法让客户"零成本试用 30 天"？
- [ ] **定价是否带来自我伤害的使用行为？** 客户为了省钱而减少使用 → 产品价值无法体现 → 次年断约

## 对产品决策者的关键启示

1. **定价模型是产品的"隐性 DNA"**：它潜移默化地塑造客户如何用你的产品——能不能用、用多少、怎么用
2. **按资产/用户定价是 SOC 产品最健康的模型**：与客户利益对齐、收入可预测、无副作用
3. **"情报内嵌"是提升感知价值的好策略**：$18K 的情报打包进 $36K 的产品，让后者看起来"性价比极高"
4. **试用/免费版必须有**：SOC 产品采购风险极高（买错了不只是浪费钱，是真的可能被黑），无风险试用是降低决策门槛最有效的手段
5. **中国市场的定价需要独立设计**：不能直接汇率换算美国定价——采购预算结构、支付意愿、竞品定价逻辑完全不同
