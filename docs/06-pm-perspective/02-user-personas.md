# SOC 产品用户画像与 JTBD

> 面向：产品经理、UX 设计师、产品负责人 — 需要理解"到底是谁在用 SOC 产品、每个人想解决什么问题"
> 解决的核心决策问题：功能优先级怎么排？为什么买单人和使用者的需求总是冲突？

## TL;DR

SOC 产品有**五类核心用户**，分为**买单人**（CISO）和**使用者**（SOC Manager、L1/L2/L3 分析师、IT 运营）。买单人关心风险可视化与合规，使用者关心效率与体验。两者需求天然错位——成功的 SOC 产品必须同时满足"向上汇报"和"向下提效"两个维度。忽视任一维度都会导致产品"有人买没人用"或"有人用没人买"。

## 核心用户画像总览

```
                    买单决策权
                        ↑
                        │
    ┌───────────────────┼───────────────────┐
    │ CISO              │                   │
    │ (最终决策者)       │                   │
    │ 关键词：风险、合规、预算    │           │
    ├───────────────────┤                   │
    │ SOC Manager       │                   │
    │ (日常决策者)       │                   │
    │ 关键词：指标、团队、SLA    │           │
    ├───────────────────┤                   │
    │ L2/L3 分析师      │   IT 运营经理     │
    │ (深度使用者)       │   (落地执行者)    │
    │ 关键词：调查、准确、关联  │ 关键词：可用性、补丁 │
    ├───────────────────┤                   │
    │ L1 分析师          │   Sysadmin       │
    │ (高频使用者)       │   (操作执行者)    │
    │ 关键词：速度、清晰、SOP  │ 关键词：自动化、简单 │
    └───────────────────┴───────────────────┘
```

## 画像 1：CISO（首席信息安全官）— 买单人

### 基本信息

| 维度 | 描述 |
|------|------|
| 典型背景 | 15-20 年 IT/安全经验，偏治理与风险管理 |
| 汇报对象 | CIO / CTO / CEO / 董事会 |
| 核心 KPI | 无重大安全事故、合规审计通过、安全预算利用率 |
| 痛点 | 说不清安全投入的 ROI、出了事要背锅、董事会不理解安全 |

### JTBD（Jobs to Be Done）

| 任务 | 场景 | 期望结果 |
|------|------|---------|
| **向董事会证明安全可控** | 季度汇报、审计前 | 一张图说清"我们有多安全"，用业务语言而非技术语言 |
| **合规审计一次性通过** | 年度审计、监管检查 | 一键生成合规报告，审计留痕完整 |
| **出了事不被追责** | 安全事件发生后 | 能证明"我们做了该做的"——有监控、有流程、有记录 |
| **证明预算花得值** | 年度预算review | 有量化对比数据（MTTD 下降 X%、自动化闭环率 Y%） |
| **不被新技术概念裹挟** | 厂商推销 AI/零信任等 | 能区分"真有用"和"厂商营销"，避免买错被追责 |

### 对产品的隐性需求

- **不要技术术语**："检测规则调优"→ CISO 想听的是"误报率下降 60%"
- **要对比数据**：上了你的产品前 vs 后，指标变化了多少
- **要行业对标**："同行在用"比"产品好用"更有说服力
- **风险叙事优于效率叙事**：讲效率提升 30% 不如讲"防止了一次可能的勒索事件"

### 采购决策中的权重

CISO 通常是**最终签字人**但不一定是**唯一决策者**。在大型企业，采购链可能是：
```
CISO 提出需求 → SOC Manager 做技术评估 → 安全架构师做技术验证 → 
CISO + CIO 联合审批 → 采购部谈商务
```

> 你的产品在技术评估阶段被 SOC Manager 否决，连 CISO 的面都见不到。反过来，CISO 认可但 SOC Manager 觉得"用了反而增加工作量"的产品也会被搁置。

## 画像 2：SOC Manager — 日常决策者

### 基本信息

| 维度 | 描述 |
|------|------|
| 典型背景 | 8-15 年安全运营经验，从 L2/L3 分析师晋升 |
| 汇报对象 | CISO / 安全总监 |
| 核心 KPI | MTTD、MTTR、SLA 达标率、团队留存率、误报率 |
| 痛点 | 告警太多处理不完、招不到人、老员工离职、向 CISO 交不出漂亮的周报 |

### JTBD

| 任务 | 场景 | 期望结果 |
|------|------|---------|
| **排班和人力规划** | 每周排班、节假日安排 | 知道每班需要多少人、AI 能顶几个人的班 |
| **确保 SLA 达标** | 每天盯告警队列 | P1/P2 告警在 SLA 内有人处理，不超时 |
| **团队能力管理** | 新人入职、老人离职、技能断层 | 新人能快速上手，关键技能不集中在某一个人身上 |
| **证明团队价值** | 周报、月报 | 有仪表盘一键导出，不用手工拼 Excel |
| **处理升级事件** | L1/L2 搞不定的复杂事件 | 快速获取事件全貌、决策支持信息 |
| **调优检测策略** | 误报增加、漏报被发现 | 知道哪些规则需要调、怎么调 |

### 产品需求权重排序

1. **仪表盘与报表**（最高优先级）：直接决定他能不能向 CISO 交差
2. **SLA 跟踪与告警**：告警超时自动提醒
3. **团队绩效可视化**：谁处理了多少、谁快的慢的
4. **新人上手路径**：产品能不能降低他的培训成本
5. **弹性扩展**：业务高峰期能不能临时增加处理能力

### 一句话总结

**SOC Manager 是产品能不能"用起来"的关键人。** 他的评价标准不是你功能多强，而是"用了之后我的指标有没有变好、我的团队有没有更轻松"。

## 画像 3：L1 分析师（Tier 1）— 高频使用者

### 基本信息

| 维度 | 描述 |
|------|------|
| 典型背景 | 0-3 年安全经验，计算机/信息安全专业应届或 IT 转岗 |
| 汇报对象 | SOC Manager / L2 分析师 |
| 核心 KPI | 告警处理量、分诊准确率、SOP 执行准确性 |
| 痛点 | 告警多到麻木、在 5-6 个工具间切换、不敢确定判断导致频繁升级 |
| 一句话描述 | "告警流水线上的工人" |

### JTBD

| 任务 | 场景 | 期望结果 |
|------|------|---------|
| **快速判断误报（FP）** | 每条告警 30 秒-2 分钟 | 系统告诉我这条为什么是 FP，我确认后一键关闭 |
| **别漏掉真阳性（TP）** | 混杂在大量 FP 中的真实威胁 | 系统帮我高亮"这条和其他不一样"，我重点看 |
| **不需要在不同工具间跳转** | 查 IP、查域名、查文件Hash、查用户行为 | 所有信息在一屏内，不切窗口 |
| **按 SOP 执行标准化动作** | 确认钓鱼邮件 → 拉黑域名、吊销会话 | 一键执行 SOP，不用手动去每个系统操作 |
| **升级时不被 L2 骂** | 把复杂事件升级给 L2 | 我提供的信息足够完整，L2 不需要从我这里重新问一遍 |

### 行为特征

- **决策时间极短**：每条告警 30 秒到几分钟，没有时间"深入研究"
- **高度依赖 SOP**：需要清晰的操作指南，不喜欢模糊判断
- **疲劳是最大敌人**：连续 4 小时看告警后判断力明显下降
- **被误报训练成"惯性关闭"**：一个规则持续产生误报，几周后他看到告警名称就直接关

### 产品设计启示

- **减少点击次数**就是提升效率——每次切换工具都有认知损耗和操作时延
- **默认操作应是最安全的选择**——L1 在疲劳时倾向于点"最方便"的按钮而非"最正确"的按钮
- **进度感和成就感至关重要**——处理了 200 条告警和"今天白干"的体验差别巨大

## 画像 4：L2/L3 高级分析师 — 深度使用者

### 基本信息

| 维度 | 描述 |
|------|------|
| 典型背景 | 3-10 年安全经验，有取证、恶意软件分析、渗透测试背景 |
| 汇报对象 | SOC Manager |
| 核心 KPI | 调查质量、误升级率、规则有效性 |
| 痛点 | 调查工具分散、上下文拼凑慢、写调查报告耗时、被 L1 升级太多低级告警 |

### JTBD

| 任务 | 场景 | 期望结果 |
|------|------|---------|
| **深度调查** | L1 升级的复杂事件 | 能自由地在多数据源间关联查询，不被预设查询模板限制 |
| **攻击链重建** | 确认入侵后还原攻击路径 | 时间线视图、自动关联 MITRE ATT&CK 技术 |
| **写调查报告** | 调查结束后 | 调查过程自动生成为报告草稿，我只需补充结论 |
| **调优检测规则** | 发现误报模式或漏报 | 从真实案例反推规则修改，测试修改前后对比 |

### 行为特征

- **不信任自动化**：L1 用 SOP，L2 的每一步都需要"自己确认过才放心"
- **工具忠诚度高**：习惯 Splunk SPL 的不会喜欢被迫换 XQL
- **需要"深潜"能力**：预设的分析路径反而是天花板——好的 L2 需要原始日志级别的访问能力
- **分享知识的方式**：写 runbook、教 L1、在 case 库里留分析记录

### 产品设计启示

- **AI 给出的结论必须可推翻**——L2 需要看到推理过程，不能只是一个分数
- **查询语言灵活性是硬需求**——不能只提供自然语言搜索，Splunk SPL/KQL 级别的查询能力是底线
- **case 库与知识沉淀**——每次调查都能沉淀为团队可复用的资产

## 画像 5：IT 运营 / Sysadmin — 落地执行者

### 基本信息

| 维度 | 描述 |
|------|------|
| 典型背景 | 系统管理员、网络工程师、云运维 |
| 汇报对象 | IT 运营经理 |
| 核心 KPI | 系统可用性、变更成功率 |
| 痛点 | SOC 总是催着打补丁但业务不让停、安全策略影响业务性能 |
| 一句话描述 | "安全动作的最终执行人，但安全不是他的首要 KPI" |

### JTBD

| 任务 | 场景 | 期望结果 |
|------|------|---------|
| **执行修复动作** | 补丁、隔离、账号禁用 | 动作清晰（具体命令/配置）、有回滚方案、操作时间窗口明确 |
| **评估修复影响** | 打补丁前 | 告知影响范围（哪些业务会中断、多久）、有无替代方案 |
| **证明自己做了** | 审计、复盘 | 操作有日志、有截图、有时间戳 |

### 产品设计启示

- **这是"被通知者"不是"主动使用者"**——他会收到 SOC 发来的工单，但不会主动打开 SOC 平台
- **让他做的事越简单越好**——最理想的情况是"一键修复 + 自动回滚"
- **不要让安全系统影响他的核心 KPI**——如果打了补丁导致业务中断，他会怪安全系统

## 用户的"两张嘴"：买单人 vs 使用者

SOC 产品最核心的 PM 挑战是 **买单人和使用者是两群需求完全不同的人**：

| 维度 | 买单人（CISO） | 使用者（分析师） |
|------|---------------|-----------------|
| 想看什么 | 仪表盘、趋势、合规报告 | 告警详情、调查工具、SOP |
| 语言偏好 | 风险、合规、ROI | 日志、规则、指标 |
| 决策标准 | 行业认可、同行在用、不会背锅 | 好不好用、省不省力、查询快不快 |
| 使用频率 | 月/周看一次 | 每天 8 小时用 |
| 付费意愿 | 愿意为"降低风险"付溢价 | 不关心价格 |

### 产品策略：两条腿走路

- **对 CISO 的产品体验**：集中在仪表盘、报表、合规审计——这些功能开发成本不高但对采购决策权重极高
- **对分析师的产品体验**：集中在告警工作台、调查工具、自动化——这是日活的基础，决定了"用不用的住"
- **连接层**：SOC Manager 是两者之间的翻译器——他需要既能看懂 CISO 关心的指标，又能理解分析师的工作流

## 对产品决策者的关键启示

1. **不要用 CISO 的视角设计分析师的界面**（反之亦然）——在功能设计阶段就要明确"这个功能是给谁用的"
2. **L1 的体验直接影响 L2 的效率**——L1 分诊准确率每提升 10%，L2 的无效工作量就下降 30%
3. **IT 运营是容易被忽视但不可或缺的用户**——如果修复动作执行不了，检测再准也没用
4. **AI 时代用户心智模型在变**：L1 从"我来看告警"变成"我来审核 AI 的判断"，界面的信息架构需要随之改变
